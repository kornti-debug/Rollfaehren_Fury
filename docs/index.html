New feature description

One connected round, from shore to shore

Assignment 2 moves the project beyond the original shooting test. The current build connects preparation, shopping, ferry movement, combat, rewards, and run progression into one repeatable loop.

01

Prepare at the dock

The ferry waits at either shore. No enemies spawn until the player deliberately starts the round.

02

Visit the shore shop

Both shop houses load one shared additive interior. Purchases and run statistics remain in the main scene.

03

Start the crossing

An interactable console inside the ferry house starts the physical trip to the opposite jetty.

04

Defend the ferry

Fish and pigeons approach during the crossing. Weapons, ferry health, money, and Wwise hooks provide feedback.

05

Choose and repeat

Arrival ends combat, opens a three-choice augment draft, and prepares the next crossing in reverse.

Technical spotlight

Self-implemented ferry movement procedure

FerryController builds a cubic Bézier route from the ferry's current dock transform to the opposite dock. This avoids a sideways linear slide while keeping movement deterministic and tunable.

  • Two dock transforms define route position and orientation.
  • Sampled route length keeps speed approximately constant along the curve.
  • The route tangent rotates the ferry bow-first during travel.
  • Platform displacement keeps the FPS controller aboard.
  • Arrival toggles the active shore and completes the gameplay round.

Character and world interactions

Player

  • Moves and looks through Input Actions
  • Fires and switches weapons
  • Uses shops and the ferry console with E

Enemies

  • Spawn throughout crossing progress
  • Fish use surface movement
  • Pigeons use full 3D movement

Ferry and economy

  • Contact damage reduces ferry health
  • Kills and arrival grant money
  • Shop upgrades modify later rounds

Feedback

  • HUD shows health, progress, money, and weapon
  • Animations communicate enemy movement
  • Wwise posts authored action events

System infrastructure

Implemented components and dependencies

The prototype uses focused MonoBehaviours coordinated by GameManager. Weapons, upgrades, and augments use ScriptableObjects so content values remain data-driven and Inspector-tunable.
